## Existing graph: `cssString` vs new pattern nodes

- **`CssStringConfig` / `CssStringNode`** (`kind: 'cssString'`): composable node for **resolved CSS snippets** with `{token.path}` substitution via `substituteTokenReferencesInString` — outputs a **scalar string** suitable for wiring into Math, Token Output, etc. It is **not** a persisted token type.
- **New `patternCss` / `patternHtml` nodes:** store **name + body** (and optional `css` for HTML) and output a **structured object** as the `values` port for **Token Output** with types `cssClass` / `htmlTemplate` / `htmlCssComponent`. Distinct purpose: **pattern library storage** aligned with table types.

## Token output path

- `evalTokenOutput` single-scalar mode already supports **non-array** `rawValues` (including objects) in `JSON.stringify([{ name, value: rawValues }])`.
- `buildTokensFromNode` currently types `pairs[].value` as `string` — must widen to **`unknown`** (or `string | PatternTokenValue`) so object values round-trip.

## Style Dictionary entry

- `buildTokens({ tokens })` accepts **nested `Record<string, unknown>`** (brands / groups). Pattern tokens must be **removed** before SD (D-09): implement a **tree walk** that drops leaf entries whose `$type` / `type` matches pattern types, or strip when flattening from `TokenGroup[]` — **single helper** + unit test.
