# Phase 999.1: CSS/HTML pattern token types — Context

**Gathered:** 2026-04-18  
**Status:** Ready for planning

<domain>
## Phase Boundary

Add **non–W3C-design-token** types used as **CSS/HTML pattern storage** in the same collection/theme model as existing tokens. These types exist so designers can store reusable CSS/HTML patterns **without** pretending they are standard W3C design tokens.

**Scope for implementation (when promoted):**

- **Table:** Up to **three** selectable token types (CSS class pattern, HTML template, HTML/CSS component) as described in the backlog — but **persisted values** consolidate into **two structural cases** (see decisions).
- **Graph:** **Two** graph node kinds only — **CSS** and **HTML** — not three separate node kinds. The HTML node can reference CSS pattern tokens and variables. The combined HTML/CSS “component” case is **not** a third graph node kind.
- **Preview:** **Deferred** in the first implementation slice; later prefer a **CodePen-style or similar** embed/wrapper in-app rather than building a minimal iframe sandbox from scratch.
- **Export:** **Omit** pattern tokens from standard Style Dictionary / default export outputs; surface **non-exported** status in the UI (badge or equivalent).

Out of scope until explicitly planned: Figma-like visual CSS library, block HTML editor, live iframe sandbox security hardening, JS/DOM tooling.

</domain>

<decisions>
## Implementation Decisions

### Value shape (two cases)

- **D-01:** Persisted `$value` / `GeneratedToken.value` uses a **small object** with at least **`name: string`** and **`body: string`**. Use one consistent pair of field names across pattern types ( **`name` + `body`** — align naming in TS and any `$value` serialization).
- **D-02:** **CSS-oriented** pattern tokens: `body` holds the **CSS** text (e.g. class rule block or pattern the user intends as “CSS class” storage).
- **D-03:** **HTML-oriented** pattern tokens: `body` holds **HTML** text. For the **HTML/CSS component** table type, add an optional **`css?: string`** on the same object so HTML + CSS live in **one token** without a third graph node kind.
- **D-04:** The three **table types** remain **discriminated by `TokenType`** (or equivalent); **graph** only instantiates **two** node kinds (CSS vs HTML). Mapping: CSS class pattern → CSS graph node; HTML template and HTML/CSS component → HTML graph node (component = same node with optional `css` populated).

### Graph

- **D-05:** **Two** React Flow node kinds for patterns: **CSS** and **HTML**. HTML nodes may connect or reference **CSS pattern tokens** (and variables) as allowed by the existing graph evaluation model — exact edge semantics left to **Claude’s discretion** in planning/implementation.

### References inside HTML

- **D-06:** Support **both** where feasible: **mustache-style** `{{path.to.token}}` **and** existing **`var(...)` / token-path** conventions used elsewhere in the app.
- **D-07:** **v1 baseline:** **freeform** HTML with **class** attributes and **CSS variables** is acceptable; full resolution in preview is **not** required for the first slice — document recommended syntax for downstream tooling.

### Preview (deferred)

- **D-08:** **Defer** built-in iframe/srcDoc preview for the first slice. When preview is added, prefer integrating a **CodePen-like or similar** external/embed wrapper pattern in-app — **Claude’s discretion** on exact library/embed and security review.

### Export

- **D-09:** **Omit** pattern tokens from standard Style Dictionary JSON/CSS/JS (and related default exports). **UI:** show a **badge** or **status** indicating **non-exported** / pattern-only tokens so users are not surprised at missing build output.

### Claude’s Discretion

- Exact TypeScript types for `TokenType` additions and narrowing `value` by type.
- Graph edge rules between HTML nodes and CSS pattern / token nodes.
- CodePen vs alternative embed; CSP and allowlists when preview ships.
- Copy and placement for the non-exported badge.
