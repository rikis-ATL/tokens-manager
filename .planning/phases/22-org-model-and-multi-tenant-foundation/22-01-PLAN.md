<objective>
Create the Organization Mongoose model and extend the User and TokenCollection schemas with a REQUIRED `organizationId` foreign-key reference per D-02. Add the compound `(organizationId, _id)` indexes per D-14 so org-scoped list queries never COLLSCAN. This is the data-layer foundation for multi-tenant isolation (TENANT-01) and the prerequisite for every other Phase 22 plan.

Purpose: Give the app a first-class `Organization` entity and the schema constraint that scopes every user and every collection to exactly one org. Without this, `assertOrgOwnership()` has nothing to compare against.

Output:
- New file `src/lib/db/models/Organization.ts` (minimal schema per D-01 — only `name` + timestamps)
- `src/lib/db/models/User.ts` extended with `organizationId: required: true` + compound index
- `src/lib/db/models/TokenCollection.ts` extended with `organizationId: required: true` + compound index
- Jest unit tests verifying schema shape and index registration

DEPLOYMENT ORDERING (per D-02 + Pitfall 3): D-02 is explicit — `organizationId` is `required: true` on both schemas. The safe deployment sequence is: run `scripts/migrate-to-org.ts` (Plan 04) against the database FIRST so every existing User and TokenCollection has an `organizationId` back-filled, THEN deploy this code with the `required: true` schema. The migration script is the mechanism that makes `required: true` safe; the schema itself does not defer enforcement. Plan 04 is therefore a deployment prerequisite, not a schema-level loosening — see its idempotency guard (D-13) and back-fill behaviour (D-12).
</objective>
