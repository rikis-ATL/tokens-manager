<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → sessionStorage | Playground edits stay in browser sessionStorage; no network write occurs when isPlayground=true |
| sessionStorage → page load | mergePlaygroundData overlays sessionStorage draft on MongoDB base at load time |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-35-02-01 | Tampering | sessionStorage | accept | sessionStorage is origin-scoped (same-origin policy). No cross-origin read is possible. Data is low-sensitivity (design tokens). |
| T-35-02-02 | Information Disclosure | sessionStorage | accept | sessionStorage cleared on tab close. No PII is stored — only token names, values, and graph coordinates. |
| T-35-02-03 | Elevation of Privilege | isPlayground guard bypass | mitigate | `isPlaygroundRef.current` is set from `col.isPlayground` which is read from the server-authenticated API response. A user cannot unilaterally set `isPlayground=false` without API access. Server-side `requireRole(WritePlayground)` is an additional safety net (D-05). |
| T-35-02-04 | Denial of Service | sessionStorage quota | accept | `savePlaygroundSession` already guards against >5MB payloads with a `console.warn` and early return. The function returns `false` on quota exceeded — no throw, no crash. |
</threat_model>
