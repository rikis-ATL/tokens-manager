**Step 1 — Create `src/mcp/tools/tokens.ts` (per D-04, D-05, D-06):**

This file exports a `registerTokenTools(server: McpServer)` function that registers 6 tools. Each tool has a Zod input schema, a description, and an async handler. Include educational inline comments explaining MCP concepts (per D-07, D-08).

Tools to register:

1. **`list_collections`** — No input params. Queries `TokenCollection.find({}, "name description _id").lean()`. Returns array of `{ id, name, description }`.

2. **`list_tokens`** — Input: `collectionId: z.string()`, optional `groupPath: z.string().optional()`. Finds collection by ID. If groupPath provided, filters tokens to that group prefix. Returns token entries as JSON.

3. **`get_token`** — Input: `collectionId: z.string()`, `tokenPath: z.string()` (dot-separated full path like `colors.brand.primary`). Looks up the token at that path in the collection's tokens object. Returns the token value and type, or isError if not found.

4. **`create_token`** — Input: `collectionId: z.string()`, `tokenPath: z.string()`, `value: z.string()`, `type: z.string().optional()` (defaults to `"color"`). Sets the token at the specified path in the collection's tokens object. Uses `findByIdAndUpdate` with `$set` on the dot-path. Returns success confirmation with the created token.

5. **`update_token`** — Input: `collectionId: z.string()`, `tokenPath: z.string()`, `value: z.string().optional()`, `type: z.string().optional()`. Updates value and/or type of an existing token. Returns updated token.

6. **`delete_token`** — Input: `collectionId: z.string()`, `tokenPath: z.string()`. Uses `$unset` on the dot-path. Returns confirmation.

Implementation details:
- Import `McpServer` type from `@modelcontextprotocol/sdk/server/mcp.js`
- Import `z` from `zod`
- Import `TokenCollection` from `@/lib/db/models/TokenCollection`
- All tool handlers wrap logic in try/catch, returning `{ content: [{ type: "text", text: ... }], isError: true }` on error
- Token paths in the MongoDB tokens object use dot notation: `tokens.colors.brand.primary.$value`
- For token value access, navigate the nested object structure — tokens follow W3C design token format where leaf tokens have `$value` and `$type` properties
- CRITICAL: Never use `console.log()` — use `console.error()` for debug output only
- Each tool description should be written for an LLM audience — explain what the tool does in natural language so Claude can choose the right tool

Educational inline comment pattern to use for each `server.registerTool()` call (mandatory per D-07/D-08):

**Step 2 — Create `src/mcp/server.ts` (per D-01, D-02, D-03):**

```typescript
#!/usr/bin/env node
/**
 * ATUI Tokens Manager — MCP Server
 *
 * This is a Model Context Protocol (MCP) server that exposes design token
 * management tools to AI assistants like Claude Desktop and Claude Code.
 *
 * MCP is a protocol that lets AI assistants discover and call tools
 * exposed by external servers. This server uses the "stdio" transport,
 * meaning it communicates via stdin/stdout using JSON-RPC messages.
 *
 * CRITICAL: Never use console.log() in this file or any file it imports.
 * stdout is the JSON-RPC communication channel. Use console.error() for
 * all debug/info output — stderr is safe and shows in the MCP client logs.
 */
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import mongoose from "mongoose";
import dbConnect from "@/lib/mongodb";

// Explicit model imports — required in standalone Node.js process
// (Mongoose models are registered lazily on first import)
import "@/lib/db/models/TokenCollection";

import { registerTokenTools } from "./tools/tokens";
import { registerGroupTools } from "./tools/groups";

const server = new McpServer({
  name: "atui-tokens-manager",
  version: "1.0.0",
});

// Register all tools before connecting
registerTokenTools(server);
registerGroupTools(server);

async function main() {
  await dbConnect();
  console.error("[MCP] MongoDB connected");

  const transport = new StdioServerTransport();
  await server.connect(transport);
  console.error("[MCP] ATUI Tokens Manager MCP server running on stdio");
}

// Clean disconnect on process exit
process.on("SIGINT", async () => {
  console.error("[MCP] Shutting down (SIGINT)...");
  await mongoose.disconnect();
  process.exit(0);
});

process.on("SIGTERM", async () => {
  console.error("[MCP] Shutting down (SIGTERM)...");
  await mongoose.disconnect();
  process.exit(0);
});

main().catch((err) => {
  console.error("[MCP] Fatal error:", err);
  process.exit(1);
});
```

Create `src/mcp/tools/groups.ts` (per D-04, D-05).

This file exports a `registerGroupTools(server: McpServer)` function that registers 2 tools:

1. **`list_groups`** — Input: `collectionId: z.string()`. Finds the collection by ID, extracts group paths from the tokens object. Token groups are determined by the nested structure of the tokens object — each non-leaf key in the tokens tree that contains sub-objects is a group. Returns an array of group path strings (e.g., `["colors", "colors.brand", "colors.neutral", "spacing"]`).

   Implementation: Walk the collection's `tokens` object recursively. A key is a group if its value is a plain object that either contains sub-objects or contains tokens (objects with `$value`). Collect all intermediate path segments.

2. **`create_group`** — Input: `collectionId: z.string()`, `groupPath: z.string()` (dot-separated, e.g., `"colors.brand.new"`), `description: z.string().optional()`. Creates an empty group at the specified path. In the W3C design token format, a group is just a nested object. Use `$set` with the dot-path to create the empty object. Returns confirmation.

   Note: If the group already exists, return a success message noting it already exists (idempotent).
